SHARING STATE BETWEEN PROMPTS AND PROGRAMS

链接：https://arxiv.org/abs/2512.14805

文章提出了共享程序状态（shared program state），提出了一个用于规范自然函数接口，并在NIGHTJAR编程系统中实现了共享程序状态。

引言

提示词实现了更高层次的抽象，因其所需实现的细节更少。

提示词与程序的互操作性：提示词与程序能操作共同的数据。

隔离程序状态（Isolated Program State）：程序员手动实现数据及广义上的程序状态在环境边界之间的传递。

共享程序状态：提示词可以直接访问程序状态，如读写程序变量、使用程序对象进行计算、在程序中实现控制流。

自然函数接口（Natural Function Interface）：规定了提示词如何与宿主程序进行交互。

论文框架将共享程序状态指定为一种自然函数接口的实现。

共享程序状态

提示词可以读写Python变量、操作对象进行计算并控制周围Python程序的执行流。

共享作用域：模型可以直接在提示词中给变量赋值<:var>。

共享堆：通过引用访问Python堆中的Python对象，如对某一字典变量就地更新。

共享控制状态：如控制流，像结束循环等。这里论文给了一个示意图，主要是通过 Python 解释器和提示词评估两个阶段进行控制流操控。 Python 解释器负责维护程序状态（值、堆、控制状态），LLM通过 Handler 取值进行评估，同时通过 Handler 像控制程序状态。

共享程序状态的自然函数接口

自然函数接口

包括以下定义：

  • 跨越提示词与程序边界的值 values，可以指定更丰富的数据类型，并通过序列化与实例化在字符串和数据类型之间转换。
  • 效果 effects，即提示词发出用于与周围代码交互的特定请求，提示词的评估结果要么是一个值，要么是一个效果。
  • 处理程序 handlers，用于提示词与程序状态的交互，Handler 能显式地选择恢复评估（让 LLM 继续评估）还是取消后续评估。

是“工具使用”（tool use）的一种泛化。

LLM需要支持以下特性：

  • 定义值的序列化/反序列化。
  • 能够设置检查点，并从检查点恢复评估；在评估取消时，对提示词进行清理。

共享程序状态作为自然函数接口

共享作用域：其效果包括查找 Lookup 和赋值 Assign。对于 Lookup，检索该变量并恢复 resume 评估；对于 Assign，在当前作用域中建立变量绑定，恢复提示词评估。

共享堆：其效果包括解引用 Deref，引用 Ref 和设置 Set。

共享状态控制：其效果为跳转 Goto，处理程序在处理此效果时不会恢复提示词评估。

系统实现

比较值得注意的是系统工程优化。系统针对 Python 进行了定制，并采取了相关优化： Break、Continue、Return、Raise 取代了通用的 Goto；为了减少 LLM 发出的效果数量，选择向 LLM 预加载变量的值和类型；缓存会存储效果追踪历史与响应的键值对。

评估

RQ1: 共享程序状态能否使程序更简洁？

RQ2: 具有共享状态的程序在性能上能否媲美人工实现？

Benchmark：论文推出了 SPSBench，包括25个程序，均使用了具有共享程序状态的提示词。

Baseline：人工实现（LLM 仅接收提示词）、人工实现（LLM 使用官方的代码解释器程序来执行）、NIGHTJAR。

值得注意的是 NIGHTJAR Baseline的准确性和效率（时间）并不好，但加入给 Python 设计的特性后弥补了这一差距。
