论文阅读：METAGPT: META PROGRAMMING FOR A MULTI-AGENT COLLABORATIVE FRAMEWORK

Created2026-04-25|Updated2026-04-26|论文阅读
|Post Views:

链接：https://arxiv.org/abs/2308.00352

引言

简单串联LLM会导致级连幻觉，复杂任务的解决方案往往会出现逻辑不一致问题。

标准操作程序（SOP）：支持任务分解，有效协调，明确成员职责，指定中间输出。

受此启发，MetaGPT要求智能体生成结构化输出，如需求文档、设计制品、流程图和接口规范，同时交接需要符合标准。作用是保持沟通的一致性，减少协作中的歧义和错误。

元编程：“以编程的方式编程（programming to program）”

相关工作

已有的基于 LLM 的智能体自动编程框架没有格式化输出模式的有效工作流程。

现有的多智能体框架聚焦于模仿人类行为，同时面临着协作的挑战。

所以论文使用 SOP 这一模仿人类行为的能格式化输出模式的工作流，解决多智能体编程协作的问题。

MetaGPT框架

架构基础

角色特化：产品经理、架构师、项目经理、工程师和 QA 工程师。为每个智能体指定名称、简介、目标、约束条件，同时为每个角色初始化上下文和特定的工具。每个智能体遵循 React 风格。

工作流：

  1. 产品经理制订产品需求文档（PRD），包括用户故事和需求池
  2. 结构化的 PRD 传给架构师，转化为系统组件设计，如文件列表、数据结构、接口定义
  3. 文档被送回产品经理进行任务分配
  4. 工程师执行指定的类和函数
  5. QA 工程师制定测试用例以确保代码质量

通信协议

使用结构化通信来规范智能体通信方式。

发布-订阅机制：智能体发布信息到信息池中，同时通过角色配置来从提取相关信息，防止广播所有信息导致的信息过载。

迭代编程

可执行反馈机制：在初始代码生成后，工程师编写并执行相应的单元测试用例，如果结果可以，那么就把代码加入到消息池中；如果不行，则继续调试和改进代码，直到测试通过或达到重试上限。

实验评估

数据集

两个公开数据集（HumanEval + MBPP）和一个自行生成的 SoftwareDev。

评估标准

对于两个公开数据集，使用 Pass@k 无偏版本公式来评估生成代码的功能的准确性：
$$
Pass@k = \mathbb{E}_{Problems}\left[1 - \frac{\binom{n-c}{k}}{\binom{n}{k}}\right]
$$
对于 SofwareDev，通过人工评估（A、E）+统计分析（B、C、D）来评估性能：

  • (A) 可执行性（Executability）：该指标将代码评分从 1（失败/无法运行）到 4（完美无瑕）。’1’ 表示无法运行，’2’ 表示可运行但不完善，’3’ 表示接近完美，’4’ 表示完美的代码。
  • (B) 成本（Cost）：成本评估包括（1）运行时间、（2）token 使用量和（3）费用。
  • (C) 代码统计（Code Statistics）：包括（1）代码文件数、（2）每文件代码行数、（3）总代码行数。
  • (D) 生产力（Productivity）：定义为 token 使用量除以代码行数，即每行代码消耗的 token 数。
  • (E) 人工修订成本（Human Revision Cost）：指手动修正代码的次数，用于解决诸如包导入错误、类名不正确或引用路径不完整等问题。通常每次修正不超过 3 行代码。
